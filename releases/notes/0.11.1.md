## 修复
- Linux 远程 bridge 发布包现在改用 Swift Static Linux SDK 构建为 `musl` 静态产物，降低旧版 SSH 主机因 `glibc` 版本过低无法启动 bridge 的风险。
- 远程 bridge 下载文件名改为 `PingIslandBridge-linux-musl-<arch>`，避免客户端或中间缓存复用旧的动态链接产物。
- 远程启动脚本只会在检测到真正依赖动态 `glibc` 的 bridge 时才尝试兼容 loader 回退，静态产物会直接执行。
- 补充 `musl` 平台导入支持，让共享 hook payload 映射代码可以在 Static Linux SDK 目标下编译。

## 说明
- Sparkle build 号提升到 `37`，版本号提升到 `0.11.1`。
- 这是基于 `0.11.0` 的远程 bridge 兼容性修复版本，重点改善旧 Linux / SSH 环境的启动可靠性，对应 issue `#95`。

## 关联 Issue

## Fixes
- Linux remote bridge release packages are now built with Swift's Static Linux SDK as `musl` static artifacts, reducing failures on older SSH hosts with older `glibc` versions.
- Remote bridge download assets now use `PingIslandBridge-linux-musl-<arch>` names so clients and intermediate caches do not reuse older dynamically linked artifacts.
- The remote launcher only attempts the compatibility loader fallback when the bridge is actually a dynamic `glibc` binary; static artifacts execute directly.
- Added `musl` import support so shared hook payload mapping code builds under the Static Linux SDK target.

## Notes
- The Sparkle build number is now `37`, and the app version is now `0.11.1`.
- This is a remote bridge compatibility patch based on `0.11.0`, focused on startup reliability for older Linux / SSH environments, matching issue `#95`.
